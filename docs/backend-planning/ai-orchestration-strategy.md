# AI Orchestration Strategy — IDEAGO (MultiGenius)

## 1. Core Design Goals

1. **Summary-first**: Users see a concise, actionable response by default. Full reasoning is on-demand only.
2. **No noisy multi-agent debate**: Agents do not simulate a conversation with each other. Each responds independently to the user query.
3. **Context efficiency**: Never send the full event log or canvas state to the LLM. Build a compressed, relevant context window.
4. **User-defined roles**: Agent behavior is shaped entirely by the user's `role_label`. No system personas are imposed.
5. **Graceful degradation**: If one agent fails, others still respond. A partial result is better than a total failure.

---

## 2. Agent Model

An agent in IDEAGO is:
- A user-defined `role_label` (free text, e.g., "UX Critic", "Structural Engineer", "Devil's Advocate")
- A reference to a project
- A context window constructed per query from the event log + canvas summary

An agent is **not**:
- A persistent stateful AI with memory across sessions (memory is reconstructed from event log)
- A hardcoded persona (no "creative agent" or "technical agent" defaults)
- An autonomous actor (agents only respond when queried)

---

## 3. Query Flow

```
User sends query
      │
      ▼
ContextBuilder
  - Reads last N events from project_events (ordered by sequence_num DESC)
  - Reads current canvas element summary (types and counts, not full JSONB)
  - Reads project metadata (title, purpose_note)
  - Reads agent's role_label
  - Assembles compressed context string (see Section 5)
      │
      ▼
Parallel LLM calls (one per requested agent)
  - Each call: system prompt + compressed context + user query
  - Max tokens per call: configurable (default 2000 output)
  - Provider: Anthropic Claude API
  - Calls are independent — no agent sees another agent's response
      │
      ▼
ResponseFormatter
  - Extracts summary (first N sentences or model-generated summary section)
  - Stores full response as full_reasoning
  - Writes to agent_responses table
      │
      ▼
API returns summary_text per agent + has_full_reasoning flag
```

---

## 4. System Prompt Structure

Each LLM call receives:

```
[SYSTEM]
You are an AI collaborator in the IDEAGO visual ideation platform.

Your role in this project: {role_label}

Project context:
- Title: {title or "Untitled project"}
- Purpose: {purpose_note or "Not specified"}
- Canvas summary: {element_summary} (e.g., "12 strokes, 2 text nodes, 1 uploaded image with 3 annotations")
- Recent activity: {compressed_event_summary}

Instructions:
- Respond from the perspective of your role.
- Be direct and specific. Avoid generic advice.
- Your response must have two parts:
  SUMMARY: [2-4 sentences maximum — this is what the user sees immediately]
  FULL REASONING: [detailed explanation — only shown if user requests it]
- Do not simulate a debate with other agents. Respond independently.
- Do not refer to yourself as an AI. Speak as the role you've been assigned.

[USER]
{user_query}
```

## 5. Context Window Construction

The context window is built fresh per query. It never includes raw JSONB canvas state.

### Compressed context components:

**1. Canvas element summary** (always included)
```
"Canvas contains: 8 pen strokes, 1 text note ('Main entrance here'), 2 uploaded images, 1 image with 4 annotations (arrows + notes)"
```
Generated by querying `canvas_elements` — counts and summaries, not raw coordinates.

**2. Recent event summary** (last N events, N = configurable, default 30)
```
"Recent activity (last 30 actions):
- Uploaded image 'floorplan.jpg' (3 days ago)
- Added annotation arrows on image
- Sent 2 queries to agents: 'what layout does this suggest?', 'is this scalable?'
- Added 5 pen strokes
- Manually saved canvas"
```
Events are formatted as human-readable sentences, not raw JSON.

**3. Prior agent responses** (last 3, summaries only)
```
"Previous agent responses (summaries):
- [Structural Engineer] Suggested load-bearing wall placement near center
- [UX Critic] Recommended wider corridor for accessibility"
```

**4. Project metadata**
```
"Project: 'Community Center Concept' | Purpose: 'Early-stage spatial planning'"
```

### What is excluded from context:
- Raw canvas JSONB (too large, not useful for LLM reasoning)
- Full text of all past agent responses (only last 3 summaries)
- Binary file content (images are referenced by description only)
- Events older than configurable threshold (default: last 7 days or last 50 events, whichever is smaller)

## 6. Token Budget

| Component | Target tokens |
|---|---|
| System prompt | ~300 |
| Canvas element summary | ~100 |
| Recent event summary | ~300 |
| Prior agent response summaries | ~200 |
| Project metadata | ~50 |
| User query | ~100 (variable) |
| **Total input** | **~1050 (target max 2000)** |
| Max output (summary + full reasoning) | 2000 |

If context exceeds 2000 input tokens: truncate event summary first (reduce N), then truncate prior response summaries.

---

## 7. Summary Extraction

The LLM is instructed to structure its response with explicit `SUMMARY:` and `FULL REASONING:` sections.

Extraction logic:
1. Parse response for `SUMMARY:` marker → extract content until `FULL REASONING:` marker
2. If markers are missing (model non-compliance): take first 3 sentences as summary
3. `summary_text` is always populated — never empty, never "No response"
4. `full_reasoning` is stored if present; `has_full_reasoning = true` in response

---

## 8. Multi-Agent Response Aggregation

- Agents respond **in parallel** (asyncio gather or Celery parallel tasks)
- Responses are **independent** — agents do not see each other's output during generation
- The API returns all responses together after all agents complete (or after per-agent timeout)
- Timeout per agent: 15 seconds. If one times out, return a timeout marker for that agent; others succeed.
- No simulated inter-agent debate output is ever constructed by the system

---

## 9. Clarification Prompt Triggers

The system may suggest a clarification prompt to the user (displayed as a soft suggestion, not a forced interruption) when:

| Condition | Suggested clarification |
|---|---|
| `purpose_note` is null + first agent query | "What are you trying to figure out with this project?" |
| Query is very short (< 5 words) + canvas is mostly empty | "Can you tell the agents a bit more about your goal?" |
| Canvas has only strokes (no text, no uploads) + 5+ agent queries sent | "Would it help to add a note or label to your sketch?" |

These are **frontend hints** — the backend provides a `clarification_hint` field in query responses when conditions are met, but the frontend decides whether and how to display it.

```json
// in POST /agents/query response:
"clarification_hint": "Your agents would benefit from knowing more about your project's purpose. Consider adding a note.",
// null if no hint
```

---

## 10. Auto-Title Generation

When called via `POST /projects/:id/auto-title`:

1. Build a short context (project metadata + first 10 events + canvas element summary)
2. LLM call with simple prompt: "Generate a concise project title (3-7 words) based on this context. Return only the title, nothing else."
3. Write result to `projects.title`, set `auto_title = true`
4. Write `project.title.auto_generated` event to event log

This is always explicitly triggered by the user or by user action (e.g., clicking "Generate Title"). Never called automatically in the background.

---

## 11. Avoiding Noisy Full Debate Output

What IDEAGO deliberately does NOT do:
- Does not construct a simulated dialogue between agents ("Agent A says X, Agent B responds Y...")
- Does not surface the full reasoning chain by default
- Does not stream intermediate thinking steps to the user
- Does not run agents sequentially where each sees the previous agent's output (creates echo chambers)

What it DOES instead:
- Each agent responds to the user's query directly and independently
- The summary is the default surface
- Full reasoning is available on-demand but never pushed
- If users want deeper debate, they can ask a follow-up question targeting specific agents

---

## 12. Cost and Risk Notes

- Each multi-agent query = N parallel LLM API calls (N = number of active agents)
- At 3 agents, each query = 3 API calls
- Mitigation: allow users to select which agents respond per query (already in API contract)
- Token costs are logged per response (`agent_responses.token_count`, `model_used`)
- No automatic retry on failure — log the failure, return error marker for that agent
- Rate limiting: apply per-user or per-project rate limits at API gateway level (not implemented in MVP, flagged in risk register)
