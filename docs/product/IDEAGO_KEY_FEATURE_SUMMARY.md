# IDEAGO Master Plan
# IDEAGO KEY FEATURE SUMMARY

## 문서 목적
이 문서는 피그마 작업 이후 현재 기준으로 실제 구현에 반영할 핵심 기능만 정리한 문서다.  
클로드코드/커서가 프론트엔드 및 백엔드 구조를 설계할 때, 이 문서를 현재 기능 기준선으로 사용한다.

---

# 1. 서비스 정의

## 서비스명
**IDEAGO**  
*(MultiGenius)*

## 한 줄 정의
IDEAGO는 사용자가 자유롭게 그리고, 쓰고, 이미지를 올리고, 아이디어를 시각적으로 정리하며, 커스터마이징 가능한 AI 에이전트와 협업해 점차 구조화된 결과물로 발전시키는 범용 시각 창조 플랫폼이다.

## 서비스 성격
- 단순 드로잉 앱이 아니다.
- 단순 코드 생성기가 아니다.
- 특정 직군 전용 앱이 아니다.
- 누구나 아이디어를 구체화하고 전달 가능한 형태로 발전시키는 도구다.

---

# 2. 현재 기준 핵심 기능

## 2-1. 새 프로젝트 생성
### 설명
사용자는 새 프로젝트를 만들고 작업을 시작할 수 있다.

### 포함 기능
- 프로젝트명 수동 입력
- 프로젝트명 자동 생성 가능
- 프로젝트 생성 후 작업 화면 진입

### 원칙
- 사용자는 처음부터 무엇을 만들지 명확하지 않을 수 있다.
- 시작 진입 장벽이 낮아야 한다.

---

## 2-2. 에이전트 구성
### 설명
사용자는 프로젝트별로 AI 에이전트 수를 정하고, 각 에이전트의 역할을 직접 정의할 수 있다.

### 포함 기능
- 에이전트 수 선택
- 에이전트 추가
- 에이전트 역할명 직접 입력
- 프로젝트 중간에 에이전트 수정/추가/삭제 가능

### 예시 역할
- 아이디어 확장
- 기획 정리
- 글쓰기 보조
- 논리/기술 검토
- 비판적 피드백
- 시각화 보조

### 원칙
- 고정된 에이전트 체계가 아니라 커스터마이징 가능한 구조여야 한다.
- 프로젝트마다 다른 에이전트 구성이 가능해야 한다.

---

## 2-3. 메인 캔버스
### 설명
IDEAGO의 중심 작업 공간. 사용자는 자유롭게 그리고, 쓰고, 아이디어를 시각적으로 정리한다.

### 포함 기능
- 자유 드로잉
- 브러시/펜
- 지우개
- 텍스트 입력
- 도형 입력
- 색상 변경
- 선택
- 이동
- 복사
- 자르기
- 확대/축소

### 원칙
- 아날로그 도화지처럼 자연스럽게 시작할 수 있어야 한다.
- 그림을 잘 못 그려도 시작할 수 있어야 한다.
- 에이전트 없이도 기본 생산성이 있어야 한다.

---

## 2-4. 이미지 업로드 + 마킹
### 설명
사용자는 현장 사진, 참고 이미지, 도면, 스크린샷 등을 업로드하고, 그 위에 직접 스케치/메모/강조 표시를 할 수 있다.

### 포함 기능
- 이미지 업로드
- 캔버스 내 배치
- 업로드 이미지 위 직접 드로잉
- 메모 작성
- 화살표/도형/강조 표시

### 활용 예시
- 인테리어 현장 사진 위 수정 요청
- 제품 이미지 위 구조 아이디어 표시
- 참고 이미지를 기반으로 대안안 제시
- 고객 요구사항 시각적 정리

### 원칙
- 백지 시작뿐 아니라 기존 자료 보완도 가능해야 한다.
- 이 기능은 보조 기능이 아니라 핵심 기능이다.

---

## 2-5. 에이전트 결과창
### 설명
사용자는 작업 중 질문을 입력할 수 있고, 에이전트는 현재 작업 맥락을 바탕으로 요약된 가이드를 제공한다.

### 포함 기능
- 사용자 질문 입력
- 에이전트 요약 응답
- 에이전트 역할 라벨 표시
- 상세보기/더보기 가능
- 필요 시 접기/펼치기

### 원칙
- 기본은 요약형 응답이다.
- 전체 실시간 토론 로그를 기본 노출하지 않는다.
- 시끄러운 다중 대화보다 정리된 결과가 우선이다.

---

## 2-6. Detail View
### 설명
사용자가 원할 때만 결과를 좀 더 정리된 형태로 확인하는 선택형 시각화 기능이다.

### 포함 기능
- Detail View 진입 버튼
- 결과 시각화 보기
- 닫기
- 필요 시 확대/비교 보기

### 원칙
- 항상 열려 있는 패널이 아니다.
- 사용하는 사람도 있고 안 쓰는 사람도 있다는 전제를 반영한다.
- 기본 화면을 복잡하게 만들지 않는다.

---

## 2-7. 프로젝트 요약 / 히스토리
### 설명
현재 프로젝트의 핵심 정보, 에이전트 구성, 작업 요약, 목적 상태, 다음 액션 등을 정리해서 보여준다.

### 포함 기능
- 프로젝트명
- 생성일 / 수정일
- 에이전트 구성 요약
- 핵심 아이디어 요약
- 사용자 메모 요약
- 작업 히스토리 요약
- 다음 추천 액션

### 원칙
- 단순 기록이 아니라 전달과 이어받기를 쉽게 해야 한다.
- 프로젝트 맥락이 끊기지 않아야 한다.

---

## 2-8. Export
### 설명
프로젝트를 구조화된 폴더/ZIP 패키지 형태로 내보내어, 개발자/디자이너/시공사/협업자에게 전달 가능하게 한다.

### 포함 대상 예시
- project-summary.md
- agents.md
- canvas-data.json
- uploads/
- visualization/
- instructions.md
- export-manifest.json

### 목적
- 단순 이미지 저장이 아니다.
- 작업 맥락과 구조를 보존한 전달 패키지여야 한다.

### 원칙
- 내보내기는 IDEAGO의 핵심 차별 요소 중 하나다.
- 개발 전달, 제작 전달, 디자인 전달, 고객-전문가 전달에 모두 활용 가능해야 한다.

---

## 2-9. 반응형 대응
### 설명
IDEAGO는 태블릿과 모바일에서 모두 사용 가능해야 한다.

### 필수 대응 상태
- 태블릿 가로
- 태블릿 세로
- 모바일 세로(9:16)

### 원칙
- UI 겹침 금지
- 화면 비율에 따라 작업 흐름이 깨지면 안 된다
- 태블릿 가로는 주 작업 기준
- 모바일은 탭형/접이식 구조 허용

---

# 3. 제품 운영 원칙

## 3-1. 사용자는 처음부터 명확하지 않을 수 있다
- IDEAGO는 완성된 질문만 받는 도구가 아니다.
- 거친 낙서, 어설픈 스케치, 미완성 아이디어도 유효해야 한다.

## 3-2. 시각화는 플랫폼 코어 기능이다
- 시각화는 에이전트 전용 기능이 아니다.
- 플랫폼 자체 기능으로 기본 제공되어야 한다.

## 3-3. 에이전트는 보조자다
- 에이전트는 아이디어를 확장, 정리, 질문, 보완한다.
- 하지만 플랫폼의 기본 작업 기능은 에이전트 없이도 작동해야 한다.

## 3-4. 사용자 행위는 구조화 데이터가 되어야 한다
- 선, 도형, 메모, 업로드, 질의, 응답, 시각화 요청, 내보내기 등은
  내부적으로 구조화된 데이터/이벤트로 저장되어야 한다.

## 3-5. Export는 전달 구조여야 한다
- 단순 파일 다운로드가 아니다.
- 작업 맥락, 에이전트 역할, 시각 자료, 다음 단계가 담긴 전달 구조여야 한다.

# 6. 클로드코드 / 커서 전달 시 해석 기준

이 문서는 현재 기준 핵심 기능선이다.  
클로드코드/커서는 이 문서를 기준으로 다음을 설계해야 한다.

- system architecture
- database schema
- API contracts
- export structure
- AI orchestration strategy
- risk register
- MVP scope

중요:
- UI 배치는 나중에 바뀔 수 있다.
- 그러나 이 문서에 적힌 기능 책임은 임의로 삭제하거나 약화하면 안 된다.
- Detail View를 상시 패널로 바꾸거나,
- Export를 단순 다운로드로 축소하거나,
- 이미지 업로드 + 마킹을 보조 기능으로 낮추면 안 된다.
